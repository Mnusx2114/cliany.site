# cliany.site 搜索任务 Case 复盘

## 一、Case 背景
这次验证目标很明确：

希望用 `cliany.site` 把“搜索任务”做通，形成一个可复用的工作流，至少包含两步：

- 执行搜索
- 读取搜索结果并结构化返回

测试对象先后用了：

- 百度
- DuckDuckGo

目标输出希望达到：

- top N 搜索结果
- 标题
- 摘要
- 链接
- 最好能直接支持后续总结

---

## 二、实际结果

### 已经证明可行的部分
`cliany.site` 在“页面操作”层面是可用的，能够比较顺畅地完成：

- 打开搜索站点
- 输入关键词
- 点击搜索按钮
- 到达结果页

这一点在百度和 DuckDuckGo 两个 case 里都基本验证过了。

### 没有打通的部分
真正关键的“结果读取”没有打通。
即使通过 `explore` 明确要求生成：

- `extract_results`
- `read_results`

并且 prompt 已明确说明需要：
- top 5
- titles
- snippets
- links
- structured data

最终生成出来的命令仍然只是：

- 再次输入
- 再次点击
- 再次跳转

并没有完成真正的数据提取。

---

## 三、核心结论

### 结论 1：当前产品能力更偏“网页操作代理”，不是“网页信息采集代理”
从结果看，`cliany.site` 的优势主要集中在：

- 页面导航
- 表单操作
- 点击流程
- 浏览器动作执行

而它的弱点则集中在：

- 页面内容读取
- 列表结构抽取
- 字段提炼
- 结构化输出

这意味着它现在更适合：
- 自动点击流程
- 重复性网页操作
- 简单工作流自动化

不太适合直接承担：
- 搜索结果采集
- 信息聚合
- 结果摘要上游的数据提取职责

---

### 结论 2：系统“成功”的定义偏执行成功，不偏任务成功
这次多次出现一种情况：

- 命令生成成功
- 命令执行成功
- 返回 `success: true`
- 返回 `status: completed`

但从用户视角，任务并没有完成。

用户要的是：
- “搜到了什么”
- “前几条是什么”
- “能否给出内容总结”

系统完成的是：
- “我已经把搜索按钮点了”

这说明当前成功标准更多在验证：
- 动作链是否可执行

而不是验证：
- 用户目标是否已达成
- 输出字段是否满足需求

这是一个非常关键的产品问题，因为它会造成**表面可用、实际不可交付**。

---

### 结论 3：命令语义与实际能力存在错配
本次生成过的命令名包括：

- `search`
- `extract_results`
- `read_results`

其中后两个命令从命名上会让人自然理解为：
“已经具备结果读取能力”。

但实际代码检查后发现，这些命令没有真正读取结果内容。

这意味着当前产品存在明显的：

- **能力命名过度乐观**
- **命令名与实际行为不一致**
- **生成结果可解释性不足**

这个问题的风险是：
- 用户或开发者会误以为能力已经生成成功
- 后续接入时才发现输出不达标
- 调试成本被后移

---

### 结论 4：`explore` 对“抽取型任务”的落地能力不稳定
这次 prompt 已经很明确，不是模糊需求，属于高约束表达：

- 指定命令名
- 指定 top 5
- 指定 title/snippet/link
- 指定 structured data
- 指定 must actually read and return content

但生成结果仍然退化成“页面交互”。

这说明问题不在 prompt 不清楚，而在产品当前对这类任务的落地机制不足。
换句话说：

**它能理解任务描述，但生成系统还不能稳定把抽取要求落到代码里。**

---

## 四、暴露出的产品短板

### 1. 抽取链路缺失
自动生成器擅长产出 action steps，但不擅长产出：

- DOM/AXTree 结果节点识别
- 重复结构遍历
- 字段映射
- JSON 返回对象构建

这说明当前生成链路里，“动作生成”比“数据采集生成”成熟得多。

### 2. 缺少任务级验收
现在更像是在检查：

- 命令有没有生成
- 页面动作能不能跑

但没有检查：

- 输出是否包含用户要求字段
- 返回值是否真正有内容
- 结果项数量是否达到 top N

导致无效命令也能被判定为“成功生成”。

### 3. 缺少能力分层
搜索任务其实至少包含两层：

- **Interaction layer**：打开页面、输入、点击
- **Extraction layer**：识别结果项、抽字段、返回结构化数据

当前 `cliany.site` 在前者强，在后者弱，但产品层没有把这两层清晰区分出来。

### 4. 对列表型页面支持不足
搜索页、本质上是标准的列表抽取页面。
如果连搜索结果页都很难稳定抽取，那么类似场景也会有同类风险：

- 电商商品列表
- 招聘职位列表
- 新闻列表
- 文档检索结果页
- CRM/后台表格列表

这说明问题可能不是搜索站点特有，而是**列表型抽取能力普遍不足**。

---

## 五、对产品定位的启发

这个 case 提醒我们，`cliany.site` 当前更适合被定位为：

### 更适合的定位
- Web workflow automation
- Browser action execution
- 可复用网页操作命令生成器

### 暂时不宜过度承诺的定位
- 通用网页信息抽取器
- 搜索结果采集器
- 结构化网页数据抓取引擎

如果产品对外表达是“能把网页工作流变成 CLI”，那目前还算成立。
但如果进一步暗示“可以稳定完成搜索+提取+总结前置数据准备”，这次 case 说明还不够稳。

---

## 六、改进建议

### 建议 1：把“读页面”做成一等能力
不要只让 LLM 自由生成 click/type。
应该补一个明确的抽取原语或抽取模板，比如：

- `extract_list`
- `extract_fields`
- `capture_visible_results`
- `return_structured_output`

让生成器不只是会动作，还会采集。

### 建议 2：引入任务验收检查
对于 `read_results` 这类命令，生成后至少做自动验收：

- 是否返回了非空 results
- 是否包含 title / url / snippet 字段
- 是否结果数 >= 3 或 >= 5
- 是否不是单纯点击动作

否则不要标记为“成功”。

### 建议 3：区分“动作命令”和“数据命令”
建议在产品层明确分成两类命令：

- Action commands
  - search
  - click_login
  - submit_form

- Data commands
  - read_results
  - extract_items
  - get_table_rows

并且对 Data commands 设置更严格的生成和校验标准。

### 建议 4：增加生成后的代码可解释性
生成结果里应直接告诉用户：

- 这个命令是“动作型”还是“抽取型”
- 它返回的字段有哪些
- 是否经过真实数据校验
- 最近一次测试返回样例是什么

这样可以显著减少“命令名看起来对，其实能力不对”的误判。

---

## 七、产品复盘结论

### 一句话版
这次 case 表明，`cliany.site` 已经能稳定完成**搜索动作自动化**，但还不能稳定完成**搜索结果抽取与结构化返回**。

### 更完整一点的结论
它当前最强的是：
- 浏览器交互自动化

它当前最弱的是：
- 结果页数据提取
- 结构化输出
- 面向用户目标的验收闭环

### 如果要给团队一个判断
可以这么说：

> `cliany.site` 现在已经证明自己是一个可用的网页动作自动化工具，但距离“可直接交付搜索任务结果”的产品形态，还差一层稳定的数据抽取与任务验收能力。
